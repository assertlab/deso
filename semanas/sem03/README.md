## Por que essa semana importa para o Sprint 0

O Sprint 0 tem um único objetivo: sair do vago para o concreto. Nas semanas 1 e 2, a equipe conheceu o stakeholder, construiu um backlog inicial e acordou como vai trabalhar junta. Agora chegou a hora de **transformar conversas em artefatos formais**.

A Semana 3 é o ponto de inflexão da fase de Composição da Mini-Sinfonia: o protótipo hi-fi será validado com o stakeholder na quinta-feira e o escopo do projeto **fechará formalmente**. A partir da Semana 4, a equipe começa a construir. Não há como construir bem o que não se entende bem — e entender bem um problema é exatamente o que Engenharia de Requisitos ensina a fazer.

---

## Objetivos da semana

Ao final desta semana, o aluno será capaz de:

- Distinguir requisitos funcionais de não funcionais e reconhecer cada tipo em descrições de sistemas reais
- Aplicar pelo menos três técnicas de elicitação de requisitos e identificar quando cada uma é mais adequada
- Escrever user stories bem formadas no formato padrão, com critérios de aceitação verificáveis
- Aplicar os critérios INVEST para avaliar a qualidade de uma user story
- Priorizar um backlog usando MoSCoW e argumentar as escolhas de prioridade
- Definir o escopo de um MVP a partir de um backlog completo

### EI2 — Exercício Individual 2: Documento de Requisitos

Dado um sistema fictício (enunciado disponível no GitHub), você deve:

1. Listar pelo menos 5 requisitos funcionais e 5 não funcionais
2. Escrever 8 user stories com critérios de aceitação no formato Gherkin (Dado/Quando/Então)
3. Priorizar o backlog usando MoSCoW, justificando cada classificação
4. Identificar os 3 atributos de qualidade mais críticos para o sistema e justificar

> Entrega via PR no repositório pessoal. Inclua no README do PR como você usou (ou não usou) IA na elaboração.
